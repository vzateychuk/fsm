## Высокоуровневая концепция AI‑assistant агента (medical adviser)

### 1. Для чего он нужен

Это интерактивный AI‑ассистент для медицинской консультации в формате диалога. Его задача — помогать пациенту структурировать жалобы и анамнез, выявлять клинически важные детали, давать понятные рекомендации и предупреждать о потенциально опасных состояниях (red flags). Ассистент ведёт консультацию непрерывно: он не "закрывает" диалог искусственно, а поддерживает последовательное уточнение выводов по мере поступления информации.

---

## 2. Ключевые процессы

### 2.1 Ingestion (индексация документов)

Система принимает медицинские документы пользователя (текстовые документы, в т.ч. Markdown), а также **изображения и сканы** (фото или скан выписки, результатов анализов и т.п.). Для немаркdown-источников перед индексацией выполняется приведение к текстовому представлению (OCR или vision-модель); далее тот же путь chunking → KB, что и для текстовых документов. Исходный файл сохраняется для прослеживаемости.

- Документ разбивается на логические фрагменты ("чанки") — небольшие смысловые блоки (например, таблица показателей, список назначений, абзац заключения).
- Для каждого чанка сохраняются структурные признаки: тип фрагмента, путь по разделам документа, заголовок, а также "поисковые теги" (нормализованные термины и алиасы), которые повышают находимость медицинских сущностей.
- После этого документ и его чанки попадают в персональную базу знаний (KB) и становятся доступными для поиска.

Детали реализации ingest для изображений и сканов — optional Phase 10 в плане реализации (roadmap), не блокируют MVP на текстовых документах.

### 2.2 Retrieval (извлечение релевантного контекста)

Retrieval‑слой превращает запрос в форму для поиска (нормализация, алиасы) и выполняет полнотекстовый ранжированный поиск по чанкам.

Результаты:
- ранжируются по релевантности,
- ограничиваются по объёму,
- группируются по документам (без "забивания" выдачи одним источником),
- сопровождаются ссылками на исходные документы.

### 2.3 Chat (консультация — агентный диалог)

Консультация — это многошаговый диалог, объединяющий историю общения, выборочные фрагменты из KB и уточняющие вопросы к пациенту.

**Инициализация сессии.** Перед первым вводом формируется системный промпт с профилем пациента (возраст, пол, хронические заболевания, препараты, аллергии) и компактным индексом документов KB (одна строка на документ: идентификатор, дата, категория, основные разделы). Индекс позволяет LLM ориентироваться в составе истории болезни без полной её загрузки в контекст.

Индекс включается в системный промпт один раз при старте сессии и не меняется в ходе диалога — это фиксированная токен-стоимость, не зависящая от длины разговора. Такой подход допустим, пока KB ограничена (MVP: единицы — десятки документов на пациента). При росте KB архитектура должна перейти к сводке в системном промпте (категории, диапазон дат, общее число документов) и предоставлять полный список через отдельный инструмент (`kb.list_documents`).

**Первый ход — baseline retrieval.** При первом сообщении пациента автоматически выполняется пассивное извлечение из KB: полнотекстовый ранжированный поиск по жалобе в заданном временном окне (query bundle) плюс первые чанки нескольких последних по дате документов вне зависимости от запроса (recency bundle). Результаты передаются в LLM как отдельный контекстный блок "Контекст из KB", явно отделённый от слов пациента, — чтобы модель не трактовала медицинские данные как часть сообщения пользователя. Baseline — оптимизация старта: ассистент получает начальный контекст без лишней задержки. Актуальность на последующих ходах обеспечивается уже не пассивным поиском, а инструментами KB, которые ассистент вызывает самостоятельно по мере необходимости.

**Агентный цикл.** Каждый ход обрабатывается внутренним циклом:

1. Сообщение пользователя добавляется в историю диалога.
2. В LLM передаётся активный контекст, сформированный политикой памяти (см. §5): резюме предыдущих ходов (если есть) + последние N реплик + результаты инструментов текущего хода + профиль пациента и индекс документов KB.
3. Если LLM запрашивает tool call, инструмент исполняется, результат добавляется в контекст и цикл повторяется.
4. Как только LLM отвечает без вызова инструментов — ответ отображается пациенту.
5. При исчерпании лимита обращений к KB за один ход все дальнейшие вызовы инструментов отклоняются с ошибкой, после чего LLM формирует финальный ответ на основе накопленного контекста.

**Инструменты KB.** LLM располагает двумя инструментами:
- `kb.search_chunks` — полнотекстовый ранжированный поиск по чанкам с фильтрами по категории и диапазону дат;
- `kb.get_document` — загрузка всех чанков конкретного документа по его идентификатору (опционально с фильтром по разделам).

Пользователь видит только ответы LLM в те моменты, когда LLM не запрашивает инструменты. Ассистент самостоятельно решает, когда и какие данные из KB нужно подтянуть.

## 3. Данные: где хранятся и откуда берутся

### 3.1 Изоляция данных: одна SQLite на пользователя

У каждого пользователя есть **собственный файл SQLite**, в котором хранятся:
- его **Knowledge Base (KB)**: документы, чанки, полнотекстовый индекс для ранжированного поиска;
- его **чат‑сессии**: история реплик, результаты tool calls, метаданные сессии;
- его **профиль**: базовая медицинская информация.

Это обеспечивает физическую изоляцию данных между пользователями и упрощает модель доступа.

### 3.2 Knowledge Base (KB)

KB — персональное хранилище медицинских документов пользователя, подготовленное для поиска. В KB хранятся:
- реестр документов (метаданные: дата, категория, источник);
- чанки (смысловые фрагменты документов);
- полнотекстовый индекс для ранжированного поиска;
- структурные признаки и поисковые теги для улучшения находимости медицинских сущностей.

### 3.3 Профиль пользователя

Базовая информация для повышения точности и безопасности консультаций:
- возраст и пол,
- аллергии,
- хронические состояния,
- текущие препараты.

### 3.4 Сессии и история диалога

Сессия — единица "чата", включающая:
- историю реплик user / assistant;
- результаты выполненных инструментов (tool results), использованных в ходе консультации;
- метаданные: название, время создания и последнего обновления, статус (активная / архивная / закреплённая).

---

## 4. Управление сессиями (memory)

Пользователь может начать консультацию, прервать её и вернуться позже, продолжив с тем же контекстом. Поведение аналогично ChatGPT / Gemini:

- **Список чатов** — все сессии пользователя, отсортированные по дате последнего обновления.
- **Переименование** — пользователь задаёт произвольный заголовок для сессии; по умолчанию заголовок формируется из первого сообщения.
- **Закрепление** — сессия фиксируется вверху списка вне зависимости от даты.
- **Архивирование** — сессия скрывается из основного списка и переносится в архив; архивные сессии доступны отдельным фильтром.
- **Восстановление контекста** — при открытии сессии в контекст LLM загружается сохранённая история (или её сжатое резюме при длинных диалогах), так что ассистент "помнит" ход предыдущей консультации.

---

## 5. Контроль контекста и компрессия

Для длинных консультаций требуется механизм, позволяющий не переполнить контекстное окно LLM при сохранении ключевых фактов:

- **Sliding window** — в активный контекст включаются N последних ходов; более старые реплики хранятся в БД, но не передаются в LLM.
- **Компрессия/резюмирование** — при достижении порога длины диалога система формирует краткое резюме накопленных клинических фактов (симптомы, гипотезы, уточнения) и заменяет им "сжатую" часть истории; итоговый контекст = резюме + последние N ходов.

Итоговый контекст, передаваемый в LLM на каждом ходе: системный промпт (профиль пациента + индекс документов KB, фиксирован) + резюме предыдущих ходов (если есть) + последние N реплик + результаты инструментов текущего хода.

**Генератор резюме.** Резюмирование выполняется отдельным компонентом системы — вне основного цикла консультации. Это важно архитектурно: основная LLM не должна прерывать диалог ради сжатия собственного контекста. Генератор резюме может использовать ту же языковую модель или облегчённую альтернативу — это деталь реализации, но компонент выделен концептуально.

## 6. Пользовательский интерфейс (фронтэнд)

Веб‑интерфейс реализует:

- **Диалоговый чат** — потоковый вывод ответов LLM, поддержка Markdown в репликах ассистента.
- **Управление сессиями** — список чатов на боковой панели с возможностью переименования, закрепления, архивирования и удаления; создание новой сессии.
- **Загрузка документов** — форма загрузки текстовых документов и **изображений/сканов** в персональную KB с отображением статуса индексации (для немаркdown-файлов — в т.ч. этап распознавания).
- **Профиль** — просмотр и редактирование базовой медицинской информации пользователя.
- **Регистрация и вход** — форма регистрации, аутентификация, восстановление пароля.

---

## 7. Регистрация и управление учётной записью

- Регистрация по email + пароль.
- После регистрации автоматически создаётся изолированная SQLite-база для пользователя.
- Ввод и редактирование профиля (возраст, пол, аллергии, хронические заболевания, препараты).
- Аутентификация с ограниченным сроком действия сессии.

## 8. Эксплуатационные требования

### 8.1 Контейнеризация

- Приложение упаковывается в контейнер; пользовательские данные монтируются как внешний том, чтобы они сохранялись между перезапусками.
- Поддерживается развёртывание одной командой, включая все зависимости.

### 8.2 Логирование

- Логи пишутся в файл с ротацией: старые файлы автоматически сжимаются и удаляются по достижению лимита.
- Уровень детализации настраивается через конфигурацию.
- В контейнерной среде логи дополнительно выводятся в стандартный поток вывода.

### 8.3 Деплой и доступ из интернета

- Приложение размещается на сервере за обратным прокси с шифрованием трафика.
- Прямой доступ к внутренним портам приложения закрыт; снаружи доступен только защищённый эндпоинт.
- Минимальный защитный слой: ограничение частоты запросов, базовые заголовки безопасности.
