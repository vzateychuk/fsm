# Извлечение данных из kb store

Ниже — как **сохранённая в SQLite информация (documents/chunks/chunks_fts + tags_text + section_path + chunk_kind)** будет использоваться на этапе **извлечения релевантных данных** (retrieval) для основного агента/медицинской LLM. Это не “извлечение показателей в структуру”, а именно **выбор релевантных фрагментов**.

## 1) Что именно мы сохранили (и зачем это нужно retrieval-слою)

### 1.1. documents (контейнер)

Хранит: - doc_type (из Target Schema ID: lab/diagnostic/consultation) - raw_text (нормализованный текст как источник правды) - source_path, sha256, audit

**Использование в retrieval:** - фильтрация по типу документа (например, lab vs diagnostic) - “показать исходник” и ссылку на файл - grouping: объединить найденные chunks по документам

### 1.2. chunks (атомарные фрагменты)

На каждый документ — много строк: - section_path (breadcrumbs: ФГДС > Результаты > Желудок > NBI...) - chunk_kind (table, list, fact, section) - text (текст чанка) - tags_text (структурные токены + алиасы, без чисел/единиц)

**Использование в retrieval:** - возвращаем не документ целиком, а **точные куски** (например таблицу ПСА, или абзац про H. pylori) - chunk_kind помогает ранжировать/фильтровать (таблицы для лабораторных запросов, рекомендации для “что назначили”) - section_path улучшает попадание: запрос “рекомендации” → сразу в нужный раздел

### 1.3. chunks_fts (FTS5 индекс)

Индексирует поля: - text - heading - section_path - tags_text

**Использование в retrieval:** - быстрый поиск по ключевым словам/фразам и по алиасам (через tags_text) - BM25-ранжирование - возможность field boosting (например, heading/section_path важнее тела)



## 2) Retrieval pipeline (после индексации)

### Шаг R0. Принять пользовательский запрос

Пример запросов: - “Где у меня упоминался H. pylori?” - “Найди все рекомендации кардиолога” - “Покажи ПТГ и связанные документы” - “Что было по венам ног — где есть рефлюкс?”

### Шаг R1. Query normalization (не NER, а техника)

Так как вы нормализуете документы в S1 (lower/NFKC/ё→е), запрос надо нормализовать **тем же способом**, иначе будет рассинхрон.

Результат: q_norm.

### Шаг R2. Query expansion через alias-map (детерминированно)

Тот же alias-map, что использовался для tags_text, применяется к запросу: - если в запросе птг → добавить паратгормон pth - если фгдс → добавить эгдс гастроскопия - если лпнп → добавить ldl ...

Важно: это не NER и не извлечение сущностей; это просто расширение по словарю.

Результат: q_fts.

### Шаг R3. FTS5 поиск по chunks

Выполняется запрос к chunks_fts с ранжированием BM25 и возможными фильтрами.

Фильтры (детерминированные): - doc_type (например, если запрос содержит “анализы”, ограничить lab; если “узи/мрт/фгдс” — diagnostic) - (опционально) временные рамки, если вы храните doc_date в documents; если не храните — без фильтра

Результат: top-K chunks (например K=10..30).

### Шаг R4. Post-ranking / grouping

После получения кандидатов: - сгруппировать по document_id - ограничить “не более N чанков на документ” (например 3–5), чтобы один документ не вытеснил остальные - можно применить простой heuristic boost: - если запрос про “рекомендации” → boost chunk_kind=list и section_path LIKE '%рекомендации%' - если запрос про “диагноз” → boost chunk_kind=list/fact и section_path LIKE '%диагноз%' - если запрос про “показатель/анализ” → boost chunk_kind=table

### Шаг R5. Возврат основному агенту

Возвращаете структуру (не обязательно сохранять в БД, это runtime): - список чанков: {doc_id, source_path, doc_type, section_path, chunk_kind, text} - плюс “evidence”: почему chunk выбран (score + совпавшие термины) — полезно для отладки

Дальше основной агент/медицинская LLM: - читает найденные чанки - делает выводы/анализ/сопоставления (это уже вне retrieval-слоя)



## 3) Как это сработает на ваших конкретных документах (типовые примеры)

### 3.1. ФГДС (2023-02-24) — “H. pylori”

В ФГДС: - есть таблица NBI-исследование с “косвенный признак ассоциации с H. pylori” - есть fact-чанк в “Клиническом значении” **h. pylori ассоциация**: ...

Запрос “h. pylori”: - FTS вернёт оба чанка, причём fact обычно будет выше, потому что текст компактнее и точнее.

### 3.2. УЗИ вен (2019-07-09) — “рефлюкс / БПВ / бедренная”

Вы разбили на: - table про параметры (рефлюкс, диаметр) - fact/list про сегменты бедра/голени

Запрос “патологический рефлюкс бпв справа”: - попадёт в table чанки “Большая подкожная вена (БПВ)” - section_path (“Правая нижняя конечность > БПВ”) даст дополнительную точность

### 3.3. Лабораторные панели — “ПТГ / ПСА / липиды”

Панели (2019-07-02, 2019-07-12, 2023-08-29, 2024-10-13, 2025-04-16) содержат: - table-чанки с показателями - list “отклонения” - fact/section клинических пояснений

Запрос “птг”: - благодаря tags_text (алиасы: паратгормон pth) найдёт и те документы, где встречается только одно написание

Запрос “пса свободный”: - попадёт в таблицу ПСА маркеры (2023-08-29)

Запрос “лпнп”: - даже если в документе “лпнп-холестерин”, токенизация tags_text добавит лпнп и лпнпхолестерин, и FTS найдёт.

### 3.4. Консультации кардиолога — “назначения / обследования / лечение”

2024-03-10_cardiologist-consultation.md и 2025-06-18_... (если добавите Target Schema ID) дают: - list-чанки “обследования”, “лекарственная терапия” - fact/section диагнозов

Запрос “что назначил кардиолог обследования”: - пост-фильтр/буст на section_path LIKE '%рекомендации%обследован%' + chunk_kind=list вернёт нужные списки.

### 3.5. МРТ поясницы (2023-01-11) — “протрузия / L4-L5 / стеноз”

Там много table и fact в клиническом значении. Запрос “l4-5 протрузия”: - попадёт в table chunk “Уровень L4-5” - даже если пользователь пишет “l4 l5”, FTS на text найдёт (в tags_text числа не включены, но числа сохраняются в chunk.text, а значит FTS по text сработает).



## 4) Какие запросы будут работать хуже именно в retrieval-слое (и почему)

1) **Семантические синонимы без алиасов**

- “цефалгия” vs “головная боль” Если нет в alias-map и нет в тексте — FTS не найдёт.

2) **Опечатки**

- FTS5 не fuzzy. Если хотите — добавляется отдельный слой query-correction (например rapidfuzz по словарю), но это отдельное решение.

3) **Чисто числовые запросы** Вы запретили числа/единицы в tags_text (что правильно), но:

- числа остаются в chunk.text, и FTS по text будет работать. Однако запрос вида “71.64” может быть нестабилен (FTS токенизация чисел зависит от tokenizer). Обычно лучше искать по названию показателя (“птг”) и дате словами.



## 5) Что нужно, чтобы retrieval был “производственным”

Минимально: - одинаковая нормализация запроса и текста (ваш S1) - alias-map (в индексации и в запросе) - бустинг по полям и типам чанков (chunk_kind, section_path) - группировка результатов по документам и ограничение чанков на документ

Если хотите, дальше могу: - предложить конкретную схему “Query → FTS5 MATCH” (с примерами MATCH-строк и bm25), - и правила field boosting/heuristics под ваши типовые вопросы.
