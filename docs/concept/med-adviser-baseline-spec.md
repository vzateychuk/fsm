# Consultation Baseline Spec — Медицинская консультация на базе KnowledgeBase (MVP)

Документ фиксирует ближайшую цель разработки: **устойчивый workflow медицинской консультации**, в котором запрос пользователя дополняется контекстом пациента из KnowledgeBase (KB) и отправляется в медицинскую LLM модель.

Контекст:
- FSM ingestion pipeline и FSM retrieval pipeline уже реализованы (см. приложенные концепты).
- Этот документ описывает слой **консультационной оркестрации** поверх KB/retrieval.

Выполнен на основании:  
- `fsm-impl-plan-ingest-concept.md`, 
- `fsm-impl-plan-retrieve-concept.md`

---

## A) Концепция системы (кратко)

### A.1 Назначение
Персональный медицинский AI-ассистент предназначен для:
- подготовки пациента к визиту к врачу,
- предварительного анализа состояния пациента на основе истории (KB),
- формирования персонализированных рекомендаций и списка вопросов к врачу,
- дальнейшего расширения (интервью, верификатор, web-search) без изменения ядра “KB → LLM”.

### A.2 Основные компоненты

#### 1) Orchestrator консультации (код)
Контрольный слой, который:
- принимает пользовательский запрос,
- запускает retrieval pipeline 1 или более раз,
- формирует структурированный KB-контекст,
- вызывает medical-LLM,
- применяет лимиты (контекст/итерации/объём данных),
- (опционально) сохраняет артефакты run (в SQLite).

Orchestrator — **не LLM модель**, а код.

#### 2) KnowledgeBase (KB)
Долговременная память пациента:
- хранит документы и чанки, построенные ingestion FSM,
- предоставляет полнотекстовый поиск (FTS5) и выдачу релевантных чанков/документов,
- скрывает детали хранения (raw_text vs filestore) как внутреннюю реализацию.

#### 3) Retrieval pipeline (уже реализован)
- нормализует запрос, расширяет алиасами,
- строит безопасный FTS запрос,
- ранжирует чанки BM25,
- может возвращать чанки и/или документы.

#### 4) Medical-LLM (внешняя модель)
- получает complaint пользователя + KB контекст,
- делает медицинский анализ, выявляет закономерности, риски, потенциальные причины,
- формирует рекомендации и вопросы к врачу.

#### 5) Будущие компоненты (не в Phase M0)
- Interview / Clarification loop (уточняющие вопросы)
- Verifier / Critic loop (контроль качества/согласованности)
- Web-search agent (доверительные источники + ссылки в ответ)
- Smart routing (RouterAgent + RouteDecision + policy enforcement)

---

## A.3 Общая логика (высокоуровнево)

### Phase M0 (Baseline)
One-shot:
- user query → baseline retrieval → KBContextBundle → 1 вызов medical-LLM → ответ

### Phase M0.1 (Agentic tools)
Инструментоподобный цикл:
- medical-LLM может запросить дополнительные данные из KB
- orchestrator выполняет retrieval и возвращает bundle
- повторный вызов medical-LLM
- итерации ограничены конфигом `max_kb_roundtrips`

---

## B) План “Baseline → Agentic” (высокоуровнево)

## B.1 Phase M0 — Baseline консультация

### Цель
Стабильно получать медицинский ответ на основе **минимального** и **контролируемого** KB контекста без сложных циклов.

### Шаги
1) принять user complaint/query
2) выполнить baseline retrieval стратегию (profile bundle - получить изначальную информацию о пациенте)
3) собрать KBContextBundle (структурированный, лимитированный, отсортированный)
4) один вызов medical-LLM с этим контекстом
5) вывести ответ пользователю

### Ожидаемый выход (M0)
- ответ с ссылками/отсылками на историю пациента (по excerpts)
- “неопределённости” (uncertainties) и “вопросы к врачу”
- без web-search
- без verifier loop

---

## B.2 Phase M0.1 — Agentic tools (запрос дополнительного контекста)

### Цель
Позволить medical-LLM запросить дополнительные фрагменты истории пациента, если baseline контекста недостаточно.

### Цикл
1) стартуем с baseline KBContextBundle как в M0
2) первый вызов medical-LLM включает:
   - baseline KB контекст
   - протокол “как запрашивать дополнительные KB данные” (см. C.6)
3) medical-LLM возвращает:
   - финальный ответ, либо
   - структурированный запрос к KB (KBQueryRequest)
4) orchestrator валидирует и трансформирует KBQueryRequest в retrieval вызовы
5) orchestrator повторно вызывает medical-LLM с дополненным контекстом
6) остановка:
   - если есть финальный ответ, или
   - если достигнут лимит итераций `max_kb_roundtrips`

### Ожидаемый выход (M0.1)
- более релевантные рекомендации из-за целевого retrieval
- trace артефактов:
  - “что модель запросила”
  - “что retrieval вернул”

## C) Ответы на вопросы проектирования (decision points)

## C.1 Лимит итераций
- Управляется конфигом: `max_kb_roundtrips`
- Рекомендованные значения:
  - M0: 0 (цикл отключён)
  - M0.1: 1 по умолчанию, но конфигурируемо (0..N)
- Требования:
  - жёсткий stop condition: не превышать лимит
  - лимиты объёма контекста на каждом roundtrip

---

## C.2 Baseline retrieval стратегия (не медицинская, “profile bundle”)
Orchestrator не должен обладать медицинской экспертизой. Baseline retrieval должен быть детерминированным и простым.

Рекомендуемая baseline стратегия:
1) **Recency bundle**
   - взять контекст из наиболее “свежих” документов (по дате/порядку ingestion)
2) **Query bundle**
   - retrieval по нормализованному user query (с алиасами)
3) Опционально: **High-signal bundle**
   - если в KB есть схематизированные типы/метаданные (лекарства/аллергии/диагнозы), то взять краткий срез
   - если нет — пропустить

MVP: начать с (1) + (2).

## C.4 Формат KBContextBundle (что именно отдаём в medical-LLM)
KBContextBundle — это артефакт, который Orchestrator формирует как структурированный текст с секциями.

Обязательные секции (M0):
1) **Key Findings** (короткие выводы в начале)
2) **KB Excerpts** (чанки, усечённые по лимиту)
3) **Provenance** (источник: document_id/path + section_path)

Правила:
- Key Findings всегда в начале (anti lost-in-the-middle)
- строгие лимиты по объёму (надо уточнить)

## C.5 Формат запроса к medical-LLM (структурированный prompt)

Вход должен быть разделён секциями:
- Key Findings
- Complaint / User question
- KB Excerpts
- Constraints / Output requirements

Выход (M0):
- текстовый ответ
- включает:
  - гипотезы с неопределённостями
  - что контролировать
  - когда обращаться к врачу срочно (red flags)
  - вопросы к врачу

## C.6 Протокол agentic запроса к KB (M0.1)

Medical-LLM должен “уметь попросить” дополнительные данные о пациенте.

KBQueryRequest (описание полей, без кода):
- intent: зачем нужны данные (какую гипотезу/проверку делает модель)
- time_window: временной интервал (например 6 месяцев)
- categories: логические категории (анализы, диагнозы, лекарства, процедуры)
- keywords: ключевые медицинские термины (Hb, ferritin, IgE, etc.)
- rationale: краткое объяснение

Orchestrator:
- валидирует запрос (allowlist categories)
- переводит в retrieval вызов(ы)
- применяет budgets (top_k/max_chars)
- возвращает результат как новый KBContextBundle

## D) Критерии готовности

### Phase M0 acceptance
- `consult` выполняет end-to-end:
  user query → retrieval → KBContextBundle → medical-LLM → ответ
- KBContextBundle соблюдает лимиты и структуру
- ответ содержит ссылки на KB контекст (по excerpts)

### Phase M0.1 acceptance
- medical-LLM может запросить дополнительные данные (KBQueryRequest)
- orchestrator выполняет минимум 1 дополнительный retrieval
- итерации ограничены `max_kb_roundtrips`
- итоговый ответ улучшается за счёт целевого контекста
