# Concept: Переход к Agent‑First KB Retrieval (с контролируемым baseline)

## 1. Контекст и проблема

### Текущее состояние (RAG‑first паттерн)
В текущей реализации retrieval выполняется “до” агента и частично подменяет его инициативу:

1. На первом сообщении пользователя сервис автоматически выполняет baseline retrieval по KB.
2. Найденные фрагменты (“чанки”) **подмешиваются** в текст пользовательского сообщения перед передачей в LLM.
3. System prompt содержит **полный** `document_index` (список всех документов в KB).

### Почему это плохо для agentic поведения
- **Избыточный retrieval**: поиск выполняется даже при приветствиях/общих вопросах.
- **Снижение автономности агента**: модель привыкает к “готовому контексту” и хуже инициирует собственные запросы к KB.
- **Негибкость**: после первого хода retrieval становится “случайным” (только если модель сама запросит tools), а baseline‑механика больше не помогает.
- **Загрязнение контекста**: полный `document_index` перегружает prompt и ухудшает качество выбора релевантных источников.

---

## 2. Цель

Перейти к **Agent‑First** подходу работы с KB, сохранив при этом качество на “холодном старте” и устойчивость:

- Агент **сам решает**, когда обращаться к KB и какими инструментами.
- Инструменты KB доступны на **каждом** turn диалога.
- Оркестратор не “подсказывает” моделью факты через подмешивание в user message.
- Retrieval и чтение документов остаются **контролируемыми** (bounded output, лимиты, прозрачные источники).
- Baseline retrieval не исчезает полностью, а становится **условным и лёгким** (light baseline), чтобы не деградировать качество.

---

## 3. Принципы целевого поведения (каноничный agentic подход)

### 3.1 Разделение “слова пациента” и “данные из KB”
- Сообщение пользователя передаётся агенту **как есть**.
- Любая информация из KB добавляется в контекст **отдельным блоком/сообщением**, чтобы модель не смешивала источник факта.

### 3.2 KB — по необходимости, но с safety‑fallback
Агент должен уметь:
- отвечать “без KB”, если вопрос общий;
- инициировать поиск, если вопрос требует фактов из истории/анализов/назначений.

Оркестратор может делать **light baseline** только когда это оправдано (см. §4).

### 3.3 Bounded output для любых инструментов
Любой инструмент KB возвращает **ограниченный по объёму** результат.
Если результат обрезан, это **явно отмечается** в ответе инструмента (truncated), чтобы агент не считал данные полными.

---

## 4. Компромисс: baseline retrieval как условный light baseline

Полный baseline retrieval “всегда и на первом ходе” заменяется на **условный** режим:

### 4.1 Когда выполняется light baseline
Light baseline выполняется только если входное сообщение пользователя похоже на “медицинский запрос”, например:
- симптомы/боль/температура/давление;
- вопросы про анализы, обследования, диагнозы;
- вопросы про лекарства/дозировки/побочные эффекты;
- “что означают мои результаты”.

Если пользователь:
- здоровается,
- задаёт общие вопросы (“как ты?”, “что ты умеешь?”),
- спрашивает не про медданные,

то baseline retrieval **не выполняется**.

### 4.2 Что входит в light baseline
Light baseline — это минимальный “контекст ориентации”, не пытающийся угадать намерение пользователя:

- **Recency digest**: несколько самых свежих документов/событий (очень кратко).
- (Опционально) **Ultra‑light query bundle**: ограниченный поиск по 1–2 ключевым терминам из запроса, но с жёстким лимитом результата.

Главная идея: дать агенту “понимание, что у пользователя есть история”, но не “залить анализы в сообщение”.

### 4.3 Как передаётся light baseline
Результаты light baseline передаются как отдельный блок “Контекст из KB”, отделённый от текста пользователя.

## 5. Целевая схема обработки сообщения (Agent‑First)

```
User → ChatService.send_message()
         │
         ├─► (опционально) LightBaselineRetriever.run()
         │        └─► context_block: "KB Context (light)"
         │
         └─► AgenticLoopRunner.run(user_message, context_block)
               │
               ├─► LLM решает: нужна ли KB?
               │     ├─► kb.search_chunks(...)
               │     ├─► kb.search_documents(...)
               │     └─► kb.get_document(...)
               │
               └─► LLM формирует ответ пользователю на основе:
                     - слов пользователя
                     - результатов инструментов (если были)
                     - контекста сессии (память/резюме)
```

---

## 6. Инструменты KB (каноничный набор)

### 6.1 `kb.search_chunks`
**Назначение:** найти релевантные фрагменты текста (чанки) по свободному запросу.

**Типовые случаи:**
- “Что было в прошлых анализах HbA1c?”
- “Покажи, что говорили врачи про головную боль”
- “Есть ли упоминания препарата X?”

**Возврат:** список чанков с источниками (document_id, дата, категория, заголовок/раздел).

---

### 6.2 `kb.search_documents` (вместо `kb_search_by_metadata(filters: dict)`)
**Назначение:** найти **документы** по метаданным и/или короткому текстовому запросу, когда пользователю нужен документ как сущность, а не фрагмент.

**Каноничная форма (строгая схема, без “свободного dict”):**
- текстовый запрос (опционально),
- фильтры: категория, диапазон дат,
- лимит,
- порядок сортировки (по релевантности/по дате).

**Типовые случаи:**
- “Найди все УЗИ за май”
- “Какие у меня были консультации кардиолога за последний год?”
- “Покажи последнее заключение эндокринолога”

**Возврат:** список документов (document_id, дата, категория, краткое описание/разделы).

> Почему не `filters: dict`: свободный dict приводит к нестабильным форматам параметров и усложняет валидацию. Для agentic tool‑calling лучше строгая схема.

---

### 6.3 `kb.get_document` (bounded / truncated)
**Назначение:** получить содержимое конкретного документа по `document_id`.

**Концептуальное поведение:**
- инструмент возвращает **весь документ в пределах допустимого объёма**;
- если документ больше допустимого объёма, результат **обрезается**, и инструмент **явно помечает**, что выдача неполная (truncated);
- опционально можно запросить только определённые разделы, но базовый режим остаётся “документ целиком (bounded)”.

**Типовые случаи:**
- “Что именно написано в документе Y?”
- “Покажи заключение целиком”
- “Нужно увидеть таблицу результатов целиком”

---

## 7. Изменения в системном промпте (концептуально)

### Было
- полный список всех документов KB в system prompt.

### Должно стать
System prompt содержит:
- роль/ограничения медицинского ассистента;
- инструкции “когда использовать KB инструменты”;
- краткие правила работы с источниками (не выдумывать факты, ссылаться на KB);
- (опционально) **компактный** каталог документов (ограниченный размером) или указание, что список документов можно получить через `kb.search_documents`.

Ключевой принцип: system prompt содержит **инструкции**, а не массив данных.

---

## 8. Ожидаемые улучшения

- Меньше “шумного” контекста на старте, лучше ответы на немедицинские сообщения.
- Агент становится реально агентным: сам инициирует retrieval и чтение документов.
- Пользователь может обращаться к истории в любой момент диалога.
- Меньше переполнений контекста благодаря bounded output и явной маркировке truncated.
- Упрощается тестирование и объяснимость: видно, когда и почему агент ходил в KB.
