# Phase M0.1 Spec — Agentic Consultation Loop

Документ описывает следующий этап разработки поверх Phase M0 (Baseline):
**управляемый агентский цикл**, в котором medical-LLM итеративно запрашивает
данные из KnowledgeBase и формирует структурированный ответ.

Выполнен на основании:
- `med-adviser-baseline-spec.md` (Phase M0 — считается завершённым)
- Обсуждения архитектурных решений (2026-05-23)

---

## A) Цель Phase M0.1

В Phase M0 (Baseline) medical-LLM получает контекст один раз и отвечает.
В Phase M0.1 LLM получает возможность **самостоятельно управлять данными**:
запрашивать нужные фрагменты истории пациента из KB и формировать
структурированный диагностический ответ.

Ключевые изменения по сравнению с M0:

1. **Агентский цикл** — LLM может сделать несколько последовательных
   запросов к KB, прежде чем вынести финальное заключение.

2. **Tool `kb.search_chunks`** — инструмент целенаправленного поиска по KB,
   которым управляет сама LLM. LLM формулирует запрос, выбирает фильтры
   и получает релевантные фрагменты.

3. **Структурированный финальный ответ** — LLM обязана выдать ответ
   в определённом формате, включающем дифференциальный диагноз,
   red flags, вопросы к пациенту и рекомендации по срочности.

Агентский цикл выполняется внутри одного вызова `consult`.
Интерактивный диалог с пациентом (ответы на вопросы в реальном времени)
относится к следующим фазам.

---

## B) Агентский цикл

### B.1 Принцип работы

LLM в каждом витке цикла принимает одно из двух решений:

- **Запросить данные из KB** — если для уточнения гипотезы нужны дополнительные
  сведения из истории пациента. LLM формулирует запрос: что искать, по каким
  категориям документов, за какой период, с каким ограничением.

- **Выдать финальный ответ** — если имеющегося контекста достаточно для
  диагностического заключения (или исчерпан лимит итераций).

В обоих случаях LLM объясняет своё решение: зачем нужны данные,
или почему контекст достаточен.

### B.2 Взаимодействие LLM — Orchestrator — KB

```
Пациент
  | жалоба
  v
Orchestrator
  |-- собирает начальный KB-контекст (как в M0)
  |-- формирует системный промпт с описанием протокола взаимодействия
  |-- передаёт контекст + жалобу в LLM
  v
LLM
  |-- анализирует контекст
  |-- решение: нужны ещё данные из KB?
  |     ДА --> возвращает запрос к KB с обоснованием
  |     НЕТ --> возвращает финальный ответ
  v
Orchestrator
  |-- если запрос к KB: выполняет поиск, добавляет результаты в контекст,
  |   возвращает управление LLM (следующий виток)
  |-- если финальный ответ: форматирует и возвращает пользователю
  |-- если достигнут лимит итераций: уведомляет LLM о лимите,
      запрашивает финальный ответ на основе имеющегося контекста
```

### B.3 Ограничения цикла

- Максимальное число дополнительных обращений к KB задаётся конфигом
  (`max_kb_roundtrips`).
- При исчерпании лимита orchestrator уведомляет LLM и требует
  финальный ответ. Незакрытые неопределённости должны быть отражены
  в поле uncertainties.
- В одном витке LLM может сделать несколько параллельных запросов к KB
  (например: одновременно запросить анализы и историю похожих диагнозов).

## C) Tool: kb.search_chunks

### C.1 Назначение

Целенаправленный поиск фрагментов (чанков) из истории пациента.
LLM формулирует запрос самостоятельно — orchestrator выполняет его
и возвращает результат.

### C.2 Что LLM может задать в запросе

- **Поисковый запрос** — медицинские термины, симптомы, показатели
  (например: "лейкоциты нейтрофилы воспаление").
- **Категория документа** — фильтр по типу документа в KB:
  Анализы, Диагноз, Консультация, Исследование, Операция, Выписка.
- **Тип чанка** — фильтр по структурному типу фрагмента:
  факт, список, таблица, секция, мета.
- **Путь секции** — фильтр по части section_path документа
  (например: "пальпация живота", "объективное обследование").
- **Временной диапазон** — дата документа от/до.
- **Количество результатов** — максимум чанков в ответе.

### C.3 Что orchestrator возвращает LLM

Список релевантных фрагментов с метаданными: идентификатор чанка,
дата документа, категория, section_path, текст и оценка релевантности
(BM25-ранг). LLM использует оценку релевантности самостоятельно:
если результаты слабые — может сделать уточняющий запрос.

Если данные не найдены — orchestrator сообщает об этом явно,
LLM продолжает на основе имеющегося контекста.

---

## D) Финальный ответ LLM

### D.1 Содержание

LLM обязана включить в финальный ответ:

- **Дифференциальный диагноз** — список гипотез с оценкой вероятности
  (высокая / средняя / низкая), аргументами за и против каждой,
  ссылками на конкретные фрагменты из KB.
- **Red flags** — признаки, при которых нужна немедленная медицинская помощь.
- **Срочность** — конкретная рекомендация: немедленно (скорая),
  сегодня (приёмный покой), плановый визит, или наблюдение дома.
- **Рекомендации** — что сделать, к какому специалисту обратиться,
  какие документы из KB взять с собой.
- **Вопросы к пациенту** — структурированный список уточнений,
  которые пациент может обдумать до визита к врачу или передать врачу.
- **Мониторинг** — что контролировать дома и когда пересмотреть оценку.
- **Неопределённости** — что неизвестно и мешает точнее сформулировать диагноз.

### D.2 Принцип

LLM не уходит в "обратитесь к врачу" без конкретики. Задача ассистента —
сформировать обоснованную гипотезу и дать пациенту понимание ситуации
до встречи с врачом. Врач не всегда доступен; ассистент должен дать
максимально конкретный и полезный ответ на основе имеющихся данных.

---

## E) Системный промпт (изменения по сравнению с M0)

Системный промпт дополняется **протоколом взаимодействия с инструментами**:

- Описание двух режимов ответа LLM: запрос данных из KB или финальный ответ.
- Описание tool `kb.search_chunks`: назначение, параметры, ограничения.
- Условия завершения цикла.
- Требования к структуре финального ответа.
- Установка на постановку конкретного диагноза (не уход в неопределённость).

---

## F) Критерии готовности (Acceptance)

- `consult` выполняет end-to-end без изменения CLI-интерфейса:
  user query → initial retrieval → agentic loop → structured final_answer → вывод
- LLM делает минимум один дополнительный запрос к KB в ходе цикла
- Цикл корректно останавливается при достижении `max_kb_roundtrips`
- Финальный ответ содержит все обязательные разделы:
  дифференциальный диагноз, red flags, срочность, вопросы к пациенту, неопределённости
- Если KB ничего не нашёл по запросу — цикл не падает, LLM продолжает
- Если LLM вернула невалидный структурированный ответ — orchestrator
  завершает gracefully с тем что есть

---

## G) Out of scope (следующие фазы)

- Интерактивный диалог с пациентом: пациент отвечает на `questions_to_patient`,
  ответы попадают в следующий виток (stateful CLI или UI)
- Web-search tool (следующий инструмент после `kb.search_chunks`)
- Verifier / Critic loop
- Улучшение алгоритмов baseline retrieval (M0 считается завершённым)

## H) Состав работ Phase M0.1

### 1. Системный промпт с протоколом инструментов

Разработать и протестировать системный промпт, который:
- объясняет LLM протокол запроса данных из KB (параметры, фильтры)
- задаёт структуру финального ответа
- формирует установку на конкретную диагностику

### 2. Structured output: схема ответа LLM

Определить и задокументировать две схемы JSON-ответа LLM:
- схема "запрос к KB" (с обоснованием и параметрами поиска)
- схема "финальный ответ" (все обязательные поля)

### 3. Orchestrator: агентский цикл

Реализовать цикл в orchestrator:
- управление историей сообщений (conversation history)
- счётчик итераций и проверка лимита
- инжект stop-message при исчерпании лимита
- парсинг structured output и ветвление (запрос KB / финальный ответ)
- graceful degradation при невалидном JSON от LLM

### 4. KB tool handler: kb.search_chunks

Реализовать обработчик tool call в orchestrator:
- маппинг параметров из запроса LLM в retrieval query
- применение фильтров (doc_category, chunk_kind, section_path, date range)
- включение relevance_score в каждый фрагмент ответа
- формирование ответа при отсутствии результатов

### 5. Форматирование вывода

Обновить CLI-вывод `consult`:
- рендеринг структурированного `final_answer` в читаемый текст
- отображение дифференциального диагноза с уровнями вероятности
- выделение red flags и срочности

### 6. Конфигурация

Добавить в конфиг параметры агентского цикла:
- `max_kb_roundtrips`
- лимит количества tool calls за один виток
- лимит количества чанков в одном tool call

### 7. Тестирование

- Unit: парсер structured output (корректный / невалидный JSON),
  маппинг параметров KB tool, stop-message при лимите
- Integration (mock LLM): сценарий одного roundtrip, сценарий лимита,
  сценарий пустого ответа KB

# 2) Что именно реализовать в Phase M0.1 (план инкрементами)

## M0.1.0 — Зафиксировать два типа ответов medical-LLM (структурированные)
Вы из spec уже хотите две схемы:
- “запрос к KB”
- “финальный ответ”

**Важно**: это должно быть JSON-only, иначе вы утонете в парсинге.

### 2.1 Schema A: KB request
Минимальные поля (достаточные для M0.1):
- `type`: `"kb_request"`
- `reason`: зачем нужны дополнительные данные
- `queries`: список запросов (может быть 1..N), каждый:
  - `query`: строка для retrieval FSM
  - `category`: опционально (из allowlist категорий)
  - `from_date`: опционально
  - `to_date`: опционально
  - `limit`: опционально (сколько чанков вернуть)
  - `limit_per_document`: опционально

### 2.2 Schema B: Final answer
Минимальные обязательные поля (из spec D):
- `type`: `"final_answer"`
- `differential`: список гипотез (name + probability high/med/low + arguments_for/against + refs)
- `urgency`: одно из: `emergency_now | today | soon | routine | self_monitor`
- `red_flags`: список
- `recommendations`: список
- `questions_for_patient`: список
- `questions_for_doctor`: список
- `uncertainties`: список

(Можно начать с менее строгой схемы, но тогда M0.1 не будет “формально” устойчивым.)

## M0.1.2 — Orchestrator loop (код)
Добавить цикл в `consult` pipeline (без изменения CLI):
- baseline KBContextBundle (как сейчас)
- вызвать medical-LLM
- если `kb_request`:
  - выполнить каждый `query` через retrieval FSM
  - сформировать “KB Tool Results” bundle (отдельная секция)
  - увеличить roundtrip counter
  - повторить
- если `final_answer`: завершить

**Guardrails:**
- `max_kb_roundtrips` из конфига
- `max_queries_per_roundtrip` (чтобы модель не запросила 20 поисков)
- лимиты выдачи на каждый tool request (`limit`, `max_total_chars`)

## M0.1.3 — Tool handler: `kb.search_chunks`
Вы не обязаны внедрять provider function-calling. Вендор-агностичный вариант:
- LLM возвращает JSON `kb_request`
- orchestrator выполняет retrieval FSM и вставляет результаты как текстовую секцию в следующий запрос

Ключ: результаты tool должны быть представлены **как structured excerpt blocks с ref** (как у вас сейчас: `doc#chunk`, `date`, `category`, `section_path`, `text`).

# 3) Почему модель сейчас “уходит в вопросы” и не даёт “конкретный диагноз”
Это правильно и безопасно. Чтобы она давала более конкретный ответ, ей нужно:
- либо больше KB контекста (M0.1),
- либо ответы пациента на уточнения (M0.2),
- плюс структурный формат “urgency + action plan”, который вы требуете в system prompt.

Сейчас вы даёте system prompt “respond with …”, но не навязываете строгую структуру и не даёте механизм дозапроса данных. Поэтому модель выбирает естественную безопасную стратегию: вопросы и предосторожности.

# 4) Что я бы поменял в baseline prompt перед стартом M0.1 (малое)
Из вашего текущего user_message видно, что туда попадают куски [7], [8] (диета/эндокринология) — они шумят.

До M0.1 лучше сразу включить rule:
- для acute complaints (боль+температура) исключать `Консультация` из recency, если она не содержит совпадений по query.

Это повысит качество tool requests: модель будет спрашивать про CRP/ОАК/УЗИ, а не про статин.

# 5) Рекомендация: что выбрать — “tools” или “вопросы пациенту” в M0.1

Вопросы пациенту оставить как поле `questions_for_patient` в финальном JSON (и выводить их), но интерактивный сбор ответов — M0.2.
