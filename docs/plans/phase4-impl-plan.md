# Phase 4 — Web Product (Single-User): План реализации

## 1. Введение

Phase 4 (см. [med-ai-adviser-roadmap.md](med-ai-adviser-roadmap.md)) переводит продукт из набора CLI-инструментов в веб-приложение для одного пользователя. Документ описывает **что и в какой последовательности** делается, какие архитектурные решения приняты и какие инварианты должны соблюдаться. Конкретные имена классов, набор полей DTO, выбор библиотек на frontend, структура YAML-конфигов и прочие детали уровня кода — определяются на этапе реализации каждого шага и не фиксируются здесь.

### Что меняется

- Продукт получает браузерный UI; CLI становится отладочным интерфейсом.
- Источником истины становится REST API. UI и любые внешние клиенты используют только его.
- Бизнес-логика, размазанная сейчас между CLI-скриптами, выносится в отдельный слой application-сервисов.
- Закладывается фундамент под будущее разделение на микросервисы — ingest и chat как независимые домены.

### Что остаётся за рамками Phase 4

- Аутентификация, multi-user, per-user БД (Phase 6–7).
- Streaming-ответы (post-Phase 4; контракт API проектируется так, чтобы расширение было неразрушающим).
- Pin-операция для сессий (post-Phase 4; БД допускает, API не экспонирует).
- Удаление документов через API (out-of-scope MVP; ошибочные загрузки чинятся операционно).
- Загрузка изображений и сканов (Phase 10 optional).
- Деплой за публичным TLS, мониторинг, метрики, distributed tracing (Phase 5+).
- Физическое разделение ingest/chat на отдельные процессы (Phase 6+).

### Существующие CLI

CLI-команды (`chat`, `ingest`, `consult`, `retrieve`) сохраняются на всю Phase 4 как debug-интерфейс. После рефакторинга они становятся тонкими обёртками над application-сервисами. Удаление CLI — отдельная задача после Phase 5.

## 2. Архитектурные решения (зафиксированы в интервью grill-me)

Каждое решение закрывает конкретную развилку дизайна.

### 2.1. API-first против CLI-first
**Решение:** API-first. REST API — основной интерфейс, CLI и UI — равноправные клиенты.
**Почему:** CLI временный; единый источник истины для будущих клиентов (мобильное, MCP, другой сервис).

### 2.2. Полная переработка против инкрементального рефакторинга
**Решение:** инкрементальный рефакторинг. Существующий код не переписывается, а декомпозируется в application-сервисы.
**Почему:** Phases 0–3 стабильны; перезапись — потеря инвестиций.

### 2.3. Streaming против синхронного ответа
**Решение:** синхронный request/response в Phase 4.
**Почему:** инфраструктурно дешевле, быстрее запускается end-to-end MVP, не блокирует переход к streaming в следующей фазе (контракт расширяется неразрушающе).

### 2.4. Состав DTO
**Решение:** минимальный набор полей. В публичном API нет токенов, latency, debug-метаданных.
**Почему:** API-контракт должен быть стабильным; всё, что не нужно UI, выносится в логи.

### 2.5. Операции над сессиями
**Решение MVP:** create, list, rename, archive, delete. Без pin, без read-only, без экспорта.

**Политика статуса в Phase 4:** БД допускает значение `pinned` (используется в CLI), но API его не экспонирует. Сериализация нормализует `pinned` к `active`. Включение pin в API — без миграции БД, расширением литерала статуса.

### 2.6. Структура репозиториев
**Решение:** два независимых репозитория — backend (текущий `fsm/`) и новый frontend.
**Почему:** независимые релизные циклы и CI/CD; контракт между ними — OpenAPI-схема.

### 2.7. Организация сервисного слоя
**Решение:** application-сервисы по доменным контекстам — chat, ingest (документы), sessions, profile. Сервисы не знают про FastAPI и про CLI; зависимости получают через конструктор.
**Почему:** эти границы соответствуют будущим микросервисам.

### 2.8. Паттерн инициализации зависимостей
**Решение:** централизованная фабрика собирает все клиенты, store-ы и сервисы. Используется lifespan-хук FastAPI; зависимости пробрасываются через DI.
**Почему:** одна точка чтения конфигов и управления жизненным циклом соединений; тестируемо.

### 2.9. Frontend-стек
**Решение:** React + Vite + TypeScript. Серверное состояние — TanStack Query, генерируемые из OpenAPI хуки. UI-компоненты — на базе Tailwind. Структура feature-sliced (`core/`, `shared/`, `features/`) с lazy-loading роутов.
**Почему:** быстрый dev-loop, богатая экосистема под чат/markdown/виртуализацию, удобная интеграция OpenAPI-кодогенерации, готовность к streaming в Phase 5+ (`ai/react`, `assistant-ui` и подобные).

### 2.10. Обработка ошибок и таймаутов
**Решение:** уже существующий retry-клиент LLM переиспользуется. На уровне API ошибки возвращаются стандартизированной JSON-схемой со стабильными машиночитаемыми кодами и осмысленными HTTP-статусами.

### 2.11. Профиль пациента
**Решение:** read-only в Phase 4. Источник — конфиг-файл. Редактирование — Phase 7.

### 2.12. Загрузка документов
**Решение:** только Markdown. Загрузка через multipart-upload. Индексация запускается асинхронно; клиент получает идентификатор и опрашивает статус. Под капотом — существующая ingest-сага.

### 2.13. CORS и доступ
**Решение:** список разрешённых origin задаётся через переменную окружения; dev-дефолт — локальный Vite-сервер.

## 3. Целевая архитектура

### 3.1. Логические слои backend

| Слой | Состав | Статус в Phase 4 |
|------|--------|------------------|
| HTTP / OpenAPI | Тонкие FastAPI-роутеры — адаптеры HTTP → application-сервисы | новый |
| Application services | Chat, Ingest, Sessions, Profile — бизнес-логика по доменным контекстам | новый |
| Domain components | Agentic loop, KB-tools, retrieval, summarizer, ingest saga, baseline retriever | существует |
| Infrastructure | Knowledge store, internal store, saga store, LLM client, file store | существует |

Нижние два слоя уже работают и протестированы. Phase 4 добавляет два верхних.

### 3.2. Доменные границы под будущие микросервисы

Phase 4 рассматривает два домена как разные сервисы, исполняемые в одном процессе:

- **Ingest-домен:** application-сервис документов, ingest-сага, write-сторона knowledge store.
- **Chat-домен:** application-сервисы чата/сессий/профиля, agentic loop, retrieval, read-сторона knowledge store, internal store, LLM, генерация саммари.

**Инварианты, обязательные в Phase 4:**

1. Ingest- и chat-сервисы не имеют взаимных импортов.
2. Все обращения к данным — через интерфейсы store-ов; прямого SQL в сервисах нет.
3. Knowledge store используется chat-доменом только на чтение, ingest-доменом — на запись.
4. Конфигурация ingest и chat живёт в разных файлах.
5. У каждого application-сервиса свой набор юнит-тестов, не зависящий от соседнего.

При переходе к физическому разделению (Phase 6+) меняется только реализация зависимостей, выдаваемых фабрикой; сами application-сервисы не меняются.

### 3.3. Структура backend-репозитория

К существующим пакетам (ingest/retrieval saga-пайплайны, store-ы, LLM-клиент, общие модули) добавляются новые: HTTP-слой (роутеры, middleware, экспорт OpenAPI), application-сервисы, фабрика инициализации, DTO-схемы.

CLI-пакет существенно сокращается — после рефакторинга команды становятся тонкими обёртками над application-сервисами.

### 3.4. Структура frontend-репозитория

Стандартная feature-sliced организация: ядро (HTTP-клиент, обработка ошибок, конфигурация), общие компоненты, фичи (chat, documents, profile, layout) с lazy-loading. Сгенерированный из OpenAPI слой клиента живёт в ядре; фичи используют узкие фасадные хуки поверх него, чтобы UI не зависел напрямую от автогенерации.

## 4. Этапность работ

Этапы выполняются последовательно: A → B → C → D → E. Внутри B шаги взаимозависимы и упорядочены; внутри C/D можно параллелить при наличии команды.

### Этап A. Подготовка

Каркас новых пакетов backend, минимальное FastAPI-приложение с health-check, добавление web-зависимостей. Никакой бизнес-логики.

**Критерий выхода:** сервер поднимается одной командой, Swagger UI отдаёт пустую схему, CI зелёный.

### Этап B. Рефакторинг — извлечение application-сервисов

Этап выполняется **до** какого-либо API-эндпоинта, чтобы каждый сервис имел тестового потребителя ещё без HTTP-слоя.

#### B1. AppContext, AppFactory и конфиг API

Централизованная фабрика собирает все store-ы и клиенты, проверяет согласованность таймаутов с retry-бюджетом LLM, инициализирует схему БД (через единый идемпотентный модуль вместо двух копий из CLI).

В этом же шаге создаётся новый файл конфигурации HTTP-слоя (таймауты, лимиты пагинации, ограничения upload, дефолтный CORS) и соответствующий Python-класс конфига по существующему в проекте паттерну.

**Критерий:** фабрика покрыта тестами, не зависит от FastAPI; повторный запуск инициализации схемы безопасен; конфиг API существует и загружается.

#### B2. Sessions service
CRUD сессий поверх internal store, без обращения к LLM/KB. Сервис нормализует БД-статус `pinned` к публичному `active`.

**Критерий:** все операции из решения 2.5 покрыты тестами.

#### B3. Profile service
Read-only выдача профиля из конфига. Минимальный сервис.

#### B4.0. Инфраструктурные правки существующего кода (отдельный PR до B4)

Изолированный шаг для всех изменений в стабильном коде Phase 0–2. Делается отдельным PR со своими тестами, чтобы не смешивать риск регрессии с разработкой нового сервиса.

В этом шаге:
- В схему БД аддитивно добавляется новая таблица для трекинга загрузок документов (статус, оригинальное имя файла, ссылка на будущий документ). Существующая таблица `documents` не меняется.
- Шаг ingest-саги, создающий запись документа, расширяется поддержкой опционального заранее выделенного идентификатора. Поведение по умолчанию (без id) сохраняется — это нужно для существующих CLI-вызовов.
- Интерфейс file store расширяется методом, дающим детерминированный путь по идентификатору без I/O.
- Появляется отдельный store для трекинга загрузок (pending/indexing/indexed/failed, тайминги, ошибка).

**Критерий:** все существующие CLI-сценарии работают без изменений; новые компоненты покрыты юнит-тестами.

#### B4. Ingest application service

Сервис собирается **поверх готовой инфраструктуры B4.0**.

**Ключевой принцип:** запись в существующей таблице `documents` создаётся только сагой (как и сейчас). Сервис её не модифицирует и не создаёт «черновых» строк. Статус процесса загрузки и оригинальное имя файла живут в новой таблице-трекере, добавленной в B4.0. Это даёт мгновенный опрашиваемый идентификатор для клиента, не трогая стабильный контракт `documents` и сохраняя существующие not-null ограничения.

Сервис:
- регистрирует upload, сохраняет файл (markdown — текст, существующий API file store достаточен), запускает сагу в фоне и возвращает идентификатор сразу;
- по успеху саги связывает upload с созданным документом и помечает его проиндексированным;
- по сбою помечает upload неудачным с сохранением причины; запись в `documents` отсутствует, частичные чанки невозможны (атомарность уже обеспечена существующим методом knowledge store);
- предоставляет операции опроса статуса и листинга, объединяя реальные индексированные документы и активные/упавшие загрузки.

**Граница на будущее:** ingest-сервис — будущий микросервис; уже сейчас он не должен импортировать ничего из chat-домена.

#### B5. Chat application service

Сервис обёртывает agentic loop, baseline retrieval, summarizer, KB-tools и persist-цикл. Воспроизводит поведение текущего CLI без stdin/stdout.

**Lifecycle agentic-loop в HTTP-режиме.** В REPL runner живёт всю сессию; в HTTP создаётся заново на каждый запрос. Существующая логика конструктора уже корректно инициализирует курсоры (save/compressed) на основании переданных истории и саммари — сервис должен лишь правильно загрузить эти данные из БД и передать в конструктор. Дублировать логику курсоров не нужно.

Persist всех сообщений хода — пакетно, **после** успешного завершения runner-а. При сбое до завершения ничего не пишется (инвариант «или весь ход, или ничего»). Компрессия применяется по существующей политике.

**Граница на будущее:** chat-сервис — будущий микросервис; не импортирует ingest-домен.

#### B6. CLI как тонкие обёртки над application-сервисами

После того как все сервисы реализованы, существующие CLI переводятся на общий AppFactory и вызовы application-сервисов:
- удаляются дубликаты инициализации схемы;
- убирается ручная сборка LLM/KB/runner-а в каждой команде;
- ingest через CLI начинает писать в тот же трекер загрузок, что и API — observability одинаковая для обоих интерфейсов;
- поведение CLI с точки зрения пользователя не меняется (флаги, выводимые сообщения).

Smoke-тесты прогоняют каждую CLI-команду на тестовых данных — ловят регрессии импорта/инициализации.

### Этап C. Backend API (FastAPI)

После B каждый сервис уже работает. На этапе C добавляются роутеры — тонкие адаптеры.

#### C1. Каркас приложения
Создание FastAPI-приложения, lifespan-хук, CORS-middleware, глобальный exception handler с маппингом доменных ошибок в стандартизированный формат, OpenAPI metadata, корреляционный middleware (request-id).

#### C2. Sessions router
CRUD-эндпоинты сессий по обычным REST-конвенциям (list, create, get, patch, delete).

#### C3. Chat router

Два эндпоинта — единичный ход чата (синхронный) и пагинированная история сообщений сессии.

**Поведение при сбоях LLM.** Ход чата атомарен: либо успех с полным ответом, либо ошибка с осмысленным кодом. Partial result не возвращается. Различимые на уровне API сценарии: успех, LLM-таймаут после исчерпания retry-бюджета, недоступность провайдера, отклонённый запрос (например, превышен лимит токенов), исчерпание roundtrips agentic-цикла, не найденная сессия. У каждого — свой машиночитаемый код и HTTP-статус.

**Retry-policy LLM.** Реализован существующим retry-клиентом. На уровне FastAPI дополнительные retry **не добавляются** — это привело бы к умножению задержек.

**Согласование HTTP-таймаута с retry-бюджетом.** Параметры HTTP-слоя (включая верхнюю границу запроса консультации) живут в конфиге API, созданном в B1. Фабрика на старте проверяет согласованность HTTP-таймаута с расчётным retry-бюджетом и предупреждает в лог при расхождении.

#### C4. Documents router
Эндпоинты загрузки, листинга и опроса статуса документа.

**Удаление документов — out of scope.** Сознательно не включается: для пилота одного пользователя ошибочные загрузки лечатся операционно; каскадное удаление (`documents` → чанки → FTS) требует отдельной операции store-а и тестового покрытия. Добавление — пост-MVP. Эта позиция фиксируется и в README API.

#### C5. Profile router
Один read-only эндпоинт.

#### C6. DTO

Минимальный набор полей по решению 2.4: сообщение (id, сессия, роль, содержимое, время), сессия (id, заголовок, статус, тайминги), запрос/ответ консультации (id ответа, текст, выполненные tool calls, цитирования, время), документ (id, имя файла, статус, категория, тайминги, причина ошибки), профиль (поля из конфига), стандартизированная ошибка (код, сообщение, опционально детали).

**Маппинги на источники данных:**
- статус документа — из таблицы-трекера загрузок; для legacy-документов из CLI до Phase 4 — `indexed`;
- имя файла — из трекера; для legacy — basename исходного пути;
- статус сессии — нормализуется к публичному множеству значений (см. 2.5);
- цитирования формируются из чанков, использованных agentic-циклом; конкретный набор полей выбирается при реализации, должен быть достаточен для рендера ссылки на источник в UI.

#### C7. Контрактные тесты и экспорт OpenAPI

Интеграционные тесты роутеров с тестовым контекстом (mock LLM, временная БД). OpenAPI-схема экспортируется в репозиторий как артефакт; CI проверяет, что закоммиченный файл совпадает с runtime-генерацией. Этот файл — единственный контракт для frontend.

**Критерий выхода этапа C:** все эндпоинты работают, OpenAPI-схема стабилизирована и закоммичена.

### Этап D. Frontend (React)

Запускается **после** этапа C — иначе frontend работает против изменяющегося контракта. Если есть выделенная команда — допустимо начать с моков, замороженных по экспортированной OpenAPI-схеме.

#### D1. Инициализация и генерация API-клиента
Скаффолд React + Vite + TypeScript. Базовый набор зависимостей (UI-стек, серверное состояние, markdown-рендер, тесты) — определяется в начале этапа D и фиксируется в `package.json`. Источник OpenAPI-схемы — выгрузка из backend-репозитория. Сгенерированный код коммитится для упрощения review и независимости CI от Java.

Фасадные хуки в фичах оборачивают сгенерированные — изолируют UI от изменений автогенерации и дают место для оптимистичных обновлений и инвалидаций кеша.

#### D2. Core слой и обработка ошибок API

Унифицированный HTTP-клиент с интерсептором, нормализующим ответ-ошибку API в типизированный класс (включая сетевые ошибки). Стратегия отображения:

- локальная ошибка действия — toast с человекочитаемым описанием и retry, где уместно;
- ошибка загрузки страницы — inline в компоненте;
- unhandled JS-ошибка — глобальный error boundary.

**Маппинг кодов ошибок API в UX-реакции** (формализуется при реализации, ниже — направление):
- LLM-таймаут / недоступность — toast «попробуйте ещё раз», текст пользователя возвращается во input;
- LLM 4xx — toast без предложения retry;
- исчерпание roundtrips — подсказка переформулировать;
- not_found — inline с редиректом;
- network_error — toast с активной кнопкой retry;
- 500 — toast с напоминанием попробовать позже; в деталях — request-id из заголовка ответа.

**Retry на стороне UI.** Серверные ошибки (LLM-таймаут, 4xx) **не ретраятся** автоматически — за это уже отвечает backend. Сетевые/транзиентные могут ретраиться ограниченно. Мутации **никогда** не ретраятся автоматически — пользователь повторяет вручную, чтобы не дублировать сообщение/upload.

**Optimistic updates.**
- Отправка пользовательского сообщения — **не оптимистичная**. Инвариант ответа консультации (либо весь ход успешен, либо ничего не записано в БД) делает оптимистичный апдейт опасным: при ошибке UI и БД разойдутся. UX: блокировка ввода + индикатор; на успех — перезапрос истории; на ошибку — toast и восстановление текста во input.
- CRUD сессий — **оптимистичные** (моментальное обновление кеша + откат при ошибке).
- Upload — **не оптимистичный** (id приходит только из ответа backend).

**Переживание перезагрузки страницы.**
- Серверное состояние пересоздаётся повторными запросами; backend — единственный источник истины.
- Текущая сессия и черновик неотправленного сообщения (по сессии) — сохраняются на клиенте локально с дебаунсом. Это безопасно: на клиенте хранится только то, что ещё не отправлено.

#### D3. Layout
Общий каркас приложения, навигация, маршрутизация с lazy-роутами.

#### D4. Chat feature
Сидбар сессий с фильтром active/archived, контекстные действия (rename/archive/delete), страница чата с историей сообщений и вводом, рендер markdown с подсветкой кода, раскрывающийся блок tool calls и цитирований у assistant-сообщений. Виртуализация списка — опционально по нагрузке.

#### D5. Documents feature
Форма загрузки (drag&drop либо нативный input — выбор фиксируется в начале D), список документов со статусом и фоновым обновлением для активно индексирующихся.

#### D6. Profile feature
Read-only страница с полями профиля и пояснением, что редактирование появится после авторизации.

#### D7. Тесты frontend

Unit-тесты — на фасадные хуки и ключевые компоненты. Smoke — `gen:api && tsc --noEmit` без ошибок.

**E2E-сценарии (обязательные для зелёного выхода D):**

| # | Сценарий |
|---|----------|
| E1 | Создать сессию, переименовать, отправить сообщение, получить ответ ассистента |
| E2 | Загрузить markdown, дождаться смены статуса на `indexed`, увидеть документ в списке |
| E3 | После E2 задать вопрос по содержимому документа — увидеть ответ с непустым блоком citations |

### Этап E. Интеграция, QA и документация

**QA:** прогон E1–E5 на реально запущенных backend + frontend; проверка CORS, таймаутов, поведения при недоступности БД; реальная индексация markdown и проверка появления его в KB-search; проверка персистентности через рестарт.

**Обязательная документация:**
- README API: как поднять backend, какие env обязательны, как запустить тесты, ссылка на Swagger UI/ReDoc.
- В README API — раздел про жизненный цикл OpenAPI-схемы (как frontend подтягивает контракт; см. §11).
- README frontend: установка, env, команды разработки/сборки/тестов/генерации API-клиента, описание структуры.

1. Backend поднимается одной командой; Swagger UI/ReDoc доступны.
2. Frontend dev-сервер поднимается одной командой.
3. Пользователь без CLI создаёт сессию, ведёт диалог, загружает markdown, видит профиль.

7. README backend и frontend написаны.

## 5. Конфигурация

Поведенческие параметры — в YAML-конфигах внутри репозитория. Секреты — только через переменные окружения, в YAML и в репозиторий не попадают. Env при необходимости перекрывают значения YAML (например, CORS-список или модель LLM).

В Phase 4 добавляется отдельный конфиг HTTP-слоя — он живёт рядом с существующими (chat, llm, retrieve, ingest и т. п.) и загружается тем же паттерном.

Имя env-переменной с ключом провайдера LLM не принципиально: ключевое — наличие OpenAI-совместимого ключа в среде. В текущем проекте используется ключ NVIDIA; смена провайдера = чтение другого имени.

В корне репозитория поддерживается образцовый `.env.example` с обязательными секретами и опциональными перекрытиями YAML; реальный `.env` — в `.gitignore`.

## 6. Миграции схемы БД

Все изменения схемы в Phase 4 — **аддитивные**: новая таблица для трекинга загрузок добавляется идемпотентно (`CREATE TABLE IF NOT EXISTS`); существующая `documents` не меняется. Применение — через единый модуль инициализации схемы, общий для FastAPI и CLI.

Полноценный механизм версионных миграций (таблица версий + раннер) **не вводится в Phase 4** — для текущих изменений он избыточен. Вводится при первом не-аддитивном изменении (ожидаемо в Phase 6 при переходе на per-user БД); тогда же выбирается, делать ли это вручную или с библиотекой.

Резервное копирование файла БД перед обновлением версии backend — операционная практика, фиксируется в README API.

## 7. Логирование и observability

Минимальный, но цельный набор требований. Полноценный observability-стек (метрики, distributed tracing, агрегация) — задача Phase 5+.

**Формат:** структурированные JSON-записи на stdout, по одной на строку. Базовые поля — время, уровень, имя события, модуль, контекст. Уровень управляется env-переменной.

**События, которые обязательно логируются** (точное имя и набор полей определяется при реализации):
- старт/конец HTTP-запроса с длительностью и request-id;
- старт/конец хода чата с количеством roundtrips и tool calls;
- каждый tool call (имя, длительность, успех);
- LLM-вызовы и retry;
- применение компрессии в чате;
- приём загрузки, шаги ingest-саги, успех/провал индексации с причиной;
- операции жизненного цикла сессий;
- инициализация схемы и загрузка конфигов.

Ошибки уровня ERROR обязательно включают полный stack trace.

**Корреляция:** request-id генерируется на каждый HTTP-запрос, прокидывается через async-контекст и автоматически попадает во все записи запроса. Возвращается клиенту в заголовке ответа — для воспроизведения проблем по логам. Идентификаторы домена (сессия, upload, документ) включаются в контекст соответствующих событий.

**Чувствительные данные** (содержимое сообщений, тексты документов) **не логируются** — только метаданные (длина, имя файла, идентификаторы).

**Хранение в Phase 4:** stdout, далее — на усмотрение запуска (terminal, `docker logs`, перенаправление в файл). Ротация и file-handlers — Phase 5.

## 8. Будущая миграция в микросервисы

Не входит в Phase 4. Документ фиксирует только инварианты (см. §3.2), которые должны соблюдаться **уже в Phase 4**, чтобы будущая миграция оставалась дешёвой.

Дорожная карта (информационно): ingest-сервис выделяется как самостоятельный процесс с REST-контрактом; chat-сервис обращается к нему вместо прямого knowledge store. Переключение делается заменой реализации зависимости в фабрике — application-сервисы не меняются.

## 11. Стратегия контракта OpenAPI

### 11.1. Решение

**Backend — code-first.** Источник истины — Pydantic-модели и сигнатуры роутеров. Файл OpenAPI-схемы — выгрузка из приложения, а не источник.

**Frontend — contract-consumer.** Типы DTO и HTTP-клиент **генерируются** из OpenAPI-схемы — выбор инструмента (например, orval в режиме TanStack-Query-хуков либо openapi-generator с Angular/React target) фиксируется в начале этапа D.

### 11.2. Почему не contract-first для обоих

Идея «единый OpenAPI-файл → код для backend и frontend» в случае FastAPI даёт минус, а не плюс:
- FastAPI спроектирован как code-first; Pydantic-модели порождают полную и точную схему. Сгенерированные из YAML Python-стабы дают более бедную типизацию и отстают по версиям.
- Появляется развилка «править код или править YAML»; на практике одна сторона рассинхронизируется.
- Для одиночного разработчика накладные расходы на синхронизацию двух источников выше выгоды.

Frontend получает все реальные плюсы контракта (типы DTO, готовый клиент, ловля расхождений на этапе компиляции) без этих минусов.

### 11.3. Поток изменения контракта

1. Backend: изменение модели/роутера → экспорт OpenAPI-схемы → коммит схемы вместе с кодом.
2. CI backend проверяет, что закоммиченная схема совпадает с runtime-генерацией.
3. После merge frontend подтягивает свежую схему и регенерирует клиент.
4. TypeScript-компилятор сразу подсветит сломанные места.

### 11.4. Альтернативы (отклонены)

- Contract-first для backend (генерация Python-стабов) — теряется главное преимущество FastAPI.
- Полностью ручные TypeScript-типы — дрейф контракта неизбежен.
- Монорепо с shared-types — не подходит к решению 2.6.
