# Medical AI Adviser — Roadmap

План реализации выведен из [концепции продукта](../concept/med-ai-adviser-concept.md).  
Ground truth — только этот документ; ниже — фазы, порядок и критерии выхода.

**Вне продуктовых фаз:** one-shot CLI `consult` — dev/smoke-инструмент для проверки KB + промптов + одного вызова LLM, без agentic-цикла и без UI. Не блокирует и не дублирует фазы roadmap.

**Отклонения от порядка разделов в concept:** до Phase 6 используется одна SQLite и один профиль (пилот на одном пользователе); [§3.1](../concept/med-ai-adviser-concept.md#31-изоляция-данных-одна-sqlite-на-пользователя) и [§7](../concept/med-ai-adviser-concept.md#7-регистрация-и-управление-учётной-записью) внедряются в Phase 6–7. Веб ([§6](../concept/med-ai-adviser-concept.md#6-пользовательский-интерфейс-фронтэнд)) — до auth, без публичной регистрации.

## Phase 0 — KB Foundation

**Концепция:** 
[§2.1](../concept/med-ai-adviser-concept.md#21-ingestion-индексация-документов), 
[§2.2](../concept/med-ai-adviser-concept.md#22-retrieval-извлечение-релевантного-контекста), 
[§3.2](../concept/med-ai-adviser-concept.md#32-knowledge-base-kb).

**Цель:** любой последующий чат опирается на стабильный ingest + retrieval.

**Scope:**
- Приём текстовых документов (в т.ч. Markdown), разбиение на чанки, теги, FTS/BM25.
- Нормализация запроса, алиасы, лимиты выдачи, группировка по документам.
- Реестр документов и чанков в SQLite (пока одна dev-БД на окружение).
- Dev/smoke: CLI `consult` (one-shot, см. блок в конце документа).

**Критерий выхода:**
- Документ индексируется end-to-end; по запросу возвращаются релевантные чанки с привязкой к источнику.

**Статус:** реализовано (pipelines ingest/retrieval).

## Phase 1 — Chat Engine (CLI)

**Концепция:** 
[§1](../concept/med-ai-adviser-concept.md#1-для-чего-он-нужен), 
[§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог).

**Цель:** движок консультации на одном пациенте / одной БД без веба и без multi-user.

**Scope (по концепции §2.3):**
- Инициализация: профиль пациента + компактный индекс документов KB в system prompt.
- Первый ход: baseline retrieval (query bundle + recency bundle), контекст KB отделён от текста пациента.
- Агентный цикл: tool calls → исполнение → повтор; ответ пациенту только без pending tools; лимит обращений к KB за ход.
- Инструменты: `kb.search_chunks`, `kb.get_document`.
- REPL: многоходовый диалог, уточняющие вопросы после завершения tool-loop на ходе.

**Критерий выхода:**
- Интерактивный `chat`: первый ход с baseline, дозапрос KB через tools, guardrails на цикл, цитирование excerpts.
- Диалог не обрывается orchestrator’ом; пациент может отвечать на уточнения в следующих ходах.

**Статус:** в основном готово (CLI `chat`, agentic loop, два KB-tool, document index).

**Остаток до закрытия фазы:** hardening (тесты agentic loop, краевые случаи пустой KB, исчерпание лимита roundtrips).

## Phase 2 — Persistent Sessions

**Концепция:** 
[§3.4](../concept/med-ai-adviser-concept.md#34-сессии-и-история-диалога), 
[§4](../concept/med-ai-adviser-concept.md#4-управление-сессиями-memory).

**Цель:** начать / прервать / продолжить консультацию с сохранённым контекстом.

**Scope:**
- Хранение в БД: реплики user/assistant, tool results, метаданные сессии (название, created/updated, active/archived/pinned).
- API или CLI: список, создание, открытие, переименование, архив, закрепление.
- Заголовок по умолчанию из первого сообщения.
- При открытии сессии — восстановление history в движок чата.

**Критерий выхода:**
- После перезапуска процесса открытая сессия продолжается с той же историей.
- Операции из [§4](../concept/med-ai-adviser-concept.md#4-управление-сессиями-memory) доступны без веба (CLI/API достаточно).

**Зависимости:** Phase 1.

## Phase 3 — Context & Compression

**Концепция:** 
[§5](../concept/med-ai-adviser-concept.md#5-контроль-контекста-и-компрессия).

**Цель:** длинные диалоги не переполняют context window; ключевые клинические факты сохраняются.

**Scope:**
- Sliding window: в LLM только последние N ходов; полная history в БД.
- Резюмирование отдельным компонентом (вне основного цикла ответа пациенту).
- Сборка контекста на ход: system (профиль + индекс) + резюме (если есть) + последние N реплик + tool results текущего хода.

**Критерий выхода:**
- Диалог с числом ходов выше порога остаётся рабочим; в промпт не попадает вся history целиком.
- Резюме обновляется по политике порога; основная LLM не «занимается сжатием» вместо ответа.

**Зависимости:** Phase 2.

## Phase 4 — Web Product (single-user)

**Концепция:** 
[§6](../concept/med-ai-adviser-concept.md#6-пользовательский-интерфейс-фронтэнд) — без регистрации и multi-tenant 
([§7](../concept/med-ai-adviser-concept.md#7-регистрация-и-управление-учётной-записью) → Phase 7).

**Цель:** продуктовый UX в браузере для одного пользователя / одной БД (пилот).

**Scope (приоритет внутри фазы):**
1. Диалоговый чат со streaming и Markdown — [§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог).
2. Управление сессиями (список, новая, переименование, pin, archive, delete) — [§4](../concept/med-ai-adviser-concept.md#4-управление-сессиями-memory).
3. Загрузка **текстовых** документов (Markdown) в KB + статус индексации — [§2.1](../concept/med-ai-adviser-concept.md#21-ingestion-индексация-документов).
4. Профиль единственного пользователя (редактирование; до Phase 6 — статический или единая запись в общей БД).

**Не в scope Phase 4:** регистрация/вход, изоляция per-user SQLite, загрузка изображений/сканов (→ Phase 10 optional).

**Критерий выхода:**
- В браузере: чат с сессиями, upload markdown, профиль; консультация соответствует [§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог) и [§5](../concept/med-ai-adviser-concept.md#5-контроль-контекста-и-компрессия) (после Phase 3).

**Зависимости:** Phase 1–3.

## Phase 5 — Operations (private pilot)

**Концепция:** 
[§8.1](../concept/med-ai-adviser-concept.md#81-контейнеризация), 
[§8.2](../concept/med-ai-adviser-concept.md#82-логирование), 
[§8.3](../concept/med-ai-adviser-concept.md#83-деплой-и-доступ-из-интернета) (ограниченно).

**Цель:** развёрнуть приложение для **тестирования одним пользователем** в закрытом контуре.

**Scope:**
- Docker, volume для данных (KB, сессии).
- Логи: файл, ротация, stdout в контейнере — [§8.2](../concept/med-ai-adviser-concept.md#82-логирование).
- Deploy за reverse proxy; **не** публичный open internet без auth (это Phase 9).
- Допустимо: localhost, VPN, IP allowlist, basic auth на прокси.

**Критерий выхода:**
- Одна команда поднимает стек; данные переживают перезапуск; пилотный пользователь стабильно работает в веб-UI.

**Зависимости:** Phase 4.

**Не путать с Phase 9:** здесь нет onboarding множества пользователей и нет обязательного 
[§7](../concept/med-ai-adviser-concept.md#7-регистрация-и-управление-учётной-записью).

## Phase 6 — Per-User Data

**Концепция:** 
[§3.1](../concept/med-ai-adviser-concept.md#31-изоляция-данных-одна-sqlite-на-пользователя), 
[§3.3](../concept/med-ai-adviser-concept.md#33-профиль-пользователя).

**Цель:** физическая изоляция KB, сессий и профиля между пользователями.

**Scope:**
- Отдельный файл SQLite на пользователя (KB + chat sessions + profile).
- Роутинг ingest/chat/retrieval по идентификатору пользователя.
- Профиль в БД пользователя (миграция с пилотного single-user режима).

**Критерий выхода:**
- Два пользователя не видят документы и чаты друг друга.
- Ingest и chat работают в контексте выбранного пользователя.

**Зависимости:** Phase 2 (схемы сессий в per-user DB), Phase 5 (пилот подтвердил стабильность).

## Phase 7 — Auth & Accounts

**Концепция:** 
[§7](../concept/med-ai-adviser-concept.md#7-регистрация-и-управление-учётной-записью).

**Цель:** регистрация, вход, привязка запросов к учётной записи и её SQLite.

**Scope:**
- Email + пароль; создание user DB при регистрации — [§3.1](../concept/med-ai-adviser-concept.md#31-изоляция-данных-одна-sqlite-на-пользователя).
- Сессии приложения с ограниченным сроком жизни.
- Регистрация и вход в веб-UI — [§6](../concept/med-ai-adviser-concept.md#6-пользовательский-интерфейс-фронтэнд).
- Редактирование профиля — [§3.3](../concept/med-ai-adviser-concept.md#33-профиль-пользователя).

**Критерий выхода:**
- Новый пользователь регистрируется → получает изолированную БД → загружает документы и ведёт чат.

**Зависимости:** Phase 6.

## Phase 8 — Large KB & Document Index

**Концепция:** 
[§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог) (абзац о росте KB и `kb.list_documents`).

**Цель:** масштабирование при десятках+ документов на пациента.

**Scope:**
- Сводка KB в system prompt (категории, диапазон дат, count) вместо полного списка.
- Инструмент `kb.list_documents` (или эквивалент) для полного каталога по запросу модели.

**Критерий выхода:**
- Старт сессии остаётся в пределах бюджета токенов при большой KB; модель может получить полный список документов через tool.

**Зависимости:** Phase 1; желательно Phase 6–7 (реальные объёмы у пользователей).

---

## Phase 9 — Internet access / Rollout

**Концепция:** 
[§8.3](../concept/med-ai-adviser-concept.md#83-деплой-и-доступ-из-интернета).

**Цель:** публичный (или широкий) доступ и подключение пользователей.

**Scope:**
- TLS, публичный endpoint, rate limit, security headers.
- Onboarding через [§7](../concept/med-ai-adviser-concept.md#7-регистрация-и-управление-учётной-записью) — **обязателен** перед открытием доступа.
- Мониторинг и эксплуатация production-like.

**Критерий выхода:**
- Внешние пользователи регистрируются и работают изолированно; нет доступа к чужим данным.

**Зависимости:** Phase 7 (и Phase 6). Phase 5 — только закрытый пилот, не замена Phase 9.

## Phase 10 (optional) — Image / scan ingest

**Концепция:** 
[§2.1](../concept/med-ai-adviser-concept.md#21-ingestion-индексация-документов), 
[§6](../concept/med-ai-adviser-concept.md#6-пользовательский-интерфейс-фронтэнд) (загрузка).

**Цель:** индексировать сканы и фото медицинских документов, не только Markdown.

**Зачем optional:** не блокирует Phase 4 (веб-MVP может стартовать с upload `.md`); для реальных данных сканы часто критичны.

**Scope:**
- Приём файлов: изображения и PDF (точный набор — при реализации).
- OCR и/или vision-LLM → текст или markdown; исходный файл в filestore.
- Интеграция в ingest saga (шаг до chunking); далее существующий pipeline.
- CLI: `ingest` принимает пути к image/PDF.
- Web: расширение формы upload (после backend), если Phase 4 уже есть.

**Критерий выхода:**
- Скан/фото → индекс в KB → находится через retrieval и 
[§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог).

**Зависимости:** Phase 0; параллельно с Phase 4–5 при необходимости; не блокирует Phase 6–9.

---

## Рекомендуемый порядок работ

1. Закрыть Phase 1 (hardening `chat`).
2. Phase 2 → Phase 3.
3. Phase 4 (веб, single-user, markdown upload).
4. Phase 5 (private pilot).
5. Phase 10 — по необходимости, параллельно с 4–5.
6. Phase 6 → Phase 7.
7. Phase 8 — по мере роста KB.
8. Phase 9 — публичный rollout.

---

## Вне roadmap (не в концепции)

Расширения вроде verifier/critic, web-search, отдельного interview-orchestrator — отдельные инициативы после стабильного продукта по [§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог).

### Dev / Smoke — `consult` (не фаза)

One-shot: жалоба → baseline retrieval → один вызов LLM → ответ.  
Назначение: быстрая проверка цепочки KB + промптов без agentic-цикла.  
Не заменяет Phase 1 и не входит в продуктовый путь по [§2.3](../concept/med-ai-adviser-concept.md#23-chat-консультация--агентный-диалог).
