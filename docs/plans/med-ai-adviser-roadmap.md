# Medical AI Adviser — Roadmap

Высокоуровеневая последовательность реализации (продуктовый уровень: что строим, зачем, критерии выхода из фазы) выведен из [концепции продукта](med-ai-adviser-concept.md).  

**Вне продуктовых фаз:** one-shot CLI `consult` — dev/smoke-инструмент для проверки KB + промптов + одного вызова LLM, без agentic-цикла и без UI. Не блокирует и не дублирует фазы roadmap.

**Отклонения от порядка разделов в concept:** до Phase 6 использовалась одна SQLite и один профиль (пилот на одном пользователе); [§3.1](см. 31-изоляция-данных-одна-sqlite-на-пользователя) и [§7](см. 7-регистрация-и-управление-учётной-записью) внедряются в Phase 6–7. Веб ([§6](см. 6-пользовательский-интерфейс-фронтэнд)) — до Phase 7 без auth и без регистрации.

**Отклонение от concept §7:** на текущем этапе учётная запись — **`username` + пароль** (не email). Email и восстановление пароля — post-Phase 7 / Phase 9. Детали: [phase6-7-impl-plan.md](phase6-7-impl-plan.md).

**Стратегия Phase 6–7:** Variant A — thin Phase 6 (data layer + migration + isolation tests), затем Phase 7 (auth, UI, profile blocker). План: [phase6-7-impl-plan.md](phase6-7-impl-plan.md).

**Порядок относительно Phase 5:** Operations (Docker, deploy в закрытом контуре) **перенесены после Phase 6–7**. Сначала multi-user data layer и auth; затем упаковка и эксплуатационный deploy. Phase 5 в таблице ниже сохраняет номер из концепции §8, но **не блокирует** Phase 6.

## Phase 0 — KB Foundation

**Концепция:** 
[§2.1](см. 21-ingestion-индексация-документов), 
[§2.2](см. 22-retrieval-извлечение-релевантного-контекста), 
[§3.2](см. 32-knowledge-base-kb).

**Цель:** любой последующий чат опирается на стабильный ingest + retrieval.

**Scope:**
- Приём текстовых документов (в т.ч. Markdown), разбиение на чанки, теги, FTS/BM25.
- Нормализация запроса, алиасы, лимиты выдачи, группировка по документам.
- Реестр документов и чанков в SQLite (пока одна dev-БД на окружение).
- Dev/smoke: CLI `consult` (one-shot, см. блок в конце документа).

**Критерий выхода:**
- Документ индексируется end-to-end; по запросу возвращаются релевантные чанки с привязкой к источнику.

**Статус:** реализовано (pipelines ingest/retrieval).

## Phase 1 — Chat Engine (CLI)

**Концепция:** 
[§1](см. 1-для-чего-он-нужен), 
[§2.3](см. 23-chat-консультация--агентный-диалог).

**Цель:** движок консультации на одном пациенте / одной БД без веба и без multi-user.

**Scope (по концепции §2.3):**
- Инициализация: профиль пациента + компактный индекс документов KB в system prompt.
- Первый ход: baseline retrieval (query bundle + recency bundle), контекст KB отделён от текста пациента.
- Агентный цикл: tool calls → исполнение → повтор; ответ пациенту только без pending tools; лимит обращений к KB за ход.
- Инструменты: `kb.search_chunks`, `kb.get_document`.
- REPL: многоходовый диалог, уточняющие вопросы после завершения tool-loop на ходе.

**Критерий выхода:**
- Интерактивный `chat`: первый ход с baseline, дозапрос KB через tools, guardrails на цикл, цитирование excerpts.
- Диалог не обрывается orchestrator’ом; пациент может отвечать на уточнения в следующих ходах.

**Статус:** в основном готово (CLI `chat`, agentic loop, два KB-tool, document index).

**Остаток до закрытия фазы:** hardening (тесты agentic loop, краевые случаи пустой KB, исчерпание лимита roundtrips).

## Phase 2 — Persistent Sessions

**Концепция:**
[§3.4](см. 34-сессии-и-история-диалога),
[§4](см. 4-управление-сессиями-memory).

**Детальный план реализации:** [phase2-3-sessions-impl.md](phase2-3-sessions-impl.md).

**Цель:** начать / прервать / продолжить консультацию с сохранённым контекстом.

**Scope:**
- Хранение в БД: реплики user/assistant, tool results, метаданные сессии (название, created/updated, active/archived/pinned).
- API или CLI: список, создание, открытие, переименование, архив, закрепление.
- Заголовок по умолчанию из первого сообщения.
- При открытии приложения, возможность начать новую сессию или загрузить одну из предыдущих сессии — восстановление history в движок чата.

**Критерий выхода:**
- После перезапуска процесса открытая сессия продолжается с той же историей.
- Операции из [§4](см. 4-управление-сессиями-memory) доступны без веба (CLI/API достаточно).

**Зависимости:** Phase 1.

**Статус:** реализовано.

## Phase 3 — Context & Compression

**Концепция:**
[§5](см. 5-контроль-контекста-и-компрессия).

**Детальный план реализации:** [phase2-3-sessions-impl.md](phase2-3-sessions-impl.md).

**Цель:** длинные диалоги не переполняют context window; ключевые клинические факты сохраняются.

**Scope:**
- Sliding window: в LLM только последние N ходов; полная history в БД.
- Резюмирование отдельным компонентом (вне основного цикла ответа пациенту).
- Сборка контекста на ход: system (профиль + индекс) + резюме (если есть) + последние N реплик + tool results текущего хода.

**Критерий выхода:**
- Диалог с числом ходов выше порога остаётся рабочим; в промпт не попадает вся history целиком.
- Резюме обновляется по политике порога; основная LLM не «занимается сжатием» вместо ответа.

**Зависимости:** Phase 2.

**Статус:** реализовано.

## Phase 4 — Web Product (single-user)

**Концепция:** 
[§6](см. 6-пользовательский-интерфейс-фронтэнд) — без регистрации и multi-tenant 
([§7](см. 7-регистрация-и-управление-учётной-записью) → Phase 7).

**Цель:** продуктовый UX в браузере для одного пользователя / одной БД (пилот).

**Scope (приоритет внутри фазы):**
1. Диалоговый чат с Markdown-рендером — [§2.3](см. 23-chat-консультация--агентный-диалог). Синхронный request/response в Phase 4; streaming перенесён в post-Phase 4 (см. ниже).
2. Управление сессиями (список, новая, переименование, archive, delete) — [§4](см. 4-управление-сессиями-memory).
3. Загрузка **текстовых** документов (Markdown) в KB + статус индексации — [§2.1](см. 21-ingestion-индексация-документов).
4. Профиль единственного пользователя (read-only из конфига; редактирование — Phase 7 после auth).

**Не в scope Phase 4:**
- регистрация/вход, изоляция per-user SQLite, загрузка изображений/сканов (→ Phase 10 optional);
- **streaming-ответ ассистента** перенесён в отдельную пост-Phase 4 задачу. Обоснование: SSE/WebSocket в FastAPI и React-стеке (с TanStack Query) — отдельная инженерная задача, которая удлиняет MVP без изменения функционального покрытия. Контракт API проектируется так, чтобы расширяться до streaming без breaking changes (см. [phase4-impl-plan.md](phase4-impl-plan.md), §2.3);
- **pin-операция** для сессий перенесена в пост-Phase 4. БД-схема (`sessions.status`) уже допускает значение `pinned` (используется в CLI), но публичный API в Phase 4 экспонирует только `active`/`archived`. Включение pin не требует изменений схемы и может быть добавлено в любой момент после Phase 4.

**Критерий выхода:**
- В браузере: чат с сессиями, upload markdown, профиль; консультация соответствует [§2.3](см. 23-chat-консультация--агентный-диалог) и [§5](см. 5-контроль-контекста-и-компрессия) (после Phase 3).

**Зависимости:** Phase 1–3.

**Детальный план реализации:** [phase4-impl-plan.md](phase4-impl-plan.md).

**Статус:** реализовано (REST API, React UI, single-user pilot).

## Phase 5 — Operations (private pilot)

**Концепция:** 
[§8.1](см. 81-контейнеризация), 
[§8.2](см. 82-логирование), 
[§8.3](см. 83-деплой-и-доступ-из-интернета) (ограниченно).

**Цель:** развёрнуть **multi-user** приложение (после Phase 6–7) для тестирования в **закрытом контуре** — Docker, volumes, логи, reverse proxy.

**Scope:**
- Docker Compose (или аналог): backend + frontend + volume `.data/db/` (`system.db`, `{username}.db`).
- Логи: файл, ротация, stdout в контейнере — [§8.2](см. 82-логирование).
- Deploy за reverse proxy; **не** публичный open internet (это Phase 9).
- Закрытый контур: VPN, IP allowlist; опционально basic auth на прокси **в дополнение** к app auth Phase 7.
- Env и runbook: `COOKIE_SECURE`, backup `.data/db/`, обновления без потери user DB.

**Критерий выхода:**
- Одна команда поднимает стек; данные переживают перезапуск; пользователи регистрируются, входят и работают в веб-UI изолированно.

**Зависимости:** Phase 7 (auth + per-user data). Phase 4 — UI/API уже есть.

**Не путать с Phase 9:** закрытый пилот с несколькими пользователями, не публичный rollout.

**Статус:** план реализации — [phase5-impl-plan.md](phase5-impl-plan.md) (profile A: HTTP/LAN; TLS/VPN — profile B). Код deploy — в работе.

## Phase 6 — Per-User Data

**Концепция:** 
[§3.1](см. 31-изоляция-данных-одна-sqlite-на-пользователя), 
[§3.3](см. 33-профиль-пользователя).

**Детальный план реализации:** [phase6-7-impl-plan.md](phase6-7-impl-plan.md) (§5).

**Цель:** физическая изоляция KB, сессий и профиля между пользователями (thin phase: data layer без auth).

**Scope:**
- Отдельный файл SQLite на пользователя (KB + chat sessions + `user_profile`).
- `UserContext` + resolver: env `USERNAME` / `DB_PATH` (pilot: `default`, `.data/db/default.db`); LRU-кэш per-user services.
- Профиль в user DB (не в `config/patient.yaml`); `ChatService` читает профиль динамически.
- One-shot миграция: `ingest.db` → `default.db` + seed из `patient.yaml`.
- CLI: `--username` + `--db`; isolation tests (dev/CI only).
- **`system.db` и auth — не в Phase 6.** Web UI без изменений (implicit `default`).

**Не в scope Phase 6:**
- login/register, `system.db`, frontend, multi-user UI;
- выбор пользователем пути к БД (флешка) — post-Phase 7; задел через `accounts.db_path`.

**Критерий выхода:**
- Два user DB (CLI/tests) не видят документы и чаты друг друга.
- Ingest и chat/API работают в контексте выбранного user DB.
- Pilot после migration: один implicit user `default`.

**Зависимости:** Phase 2 (схемы сессий в per-user DB), Phase 4 (веб/API для thin pilot). Phase 5 **не** блокирует Phase 6.

## Phase 7 — Auth & Accounts

**Концепция:** 
[§7](см. 7-регистрация-и-управление-учётной-записью) — с отклонением: **username + пароль**, не email.

**Детальный план реализации:** [phase6-7-impl-plan.md](phase6-7-impl-plan.md) (§6).

**Цель:** регистрация, вход, привязка запросов к учётной записи и её SQLite; единая модель для всех пользователей.

**Scope:**
- **`system.db`** (`.data/db/system.db`): `accounts(username, password_hash, db_path, created_at)`, `auth_sessions`.
- **Username + пароль** (argon2id, min 8); open registration (на dev — localhost; **сетевой** deploy и VPN/allowlist — Phase 5).
- Server-side session (HttpOnly cookie); **без idle-timeout** — сессия до явного logout.
- При register: `db_path = .data/db/{username}.db`, пустой профиль; auto-login.
- Protected API: `/api/v1/*` только с cookie; публично — health + `/api/v1/auth/*`.
- **Profile blocker:** chat/documents/sessions недоступны, пока профиль не заполнен (`name`, `age`, `sex`, `date_of_birth`).
- `PATCH /api/v1/profile`; login/register в веб-UI — [§6](см. 6-пользовательский-интерфейс-фронтэнд).

**Deploy (breaking):** pilot `default.db` удаляется; пользователи создаются через Register; документы re-ingest. Runbook: [phase6-7-impl-plan.md](phase6-7-impl-plan.md) §6.

**Не в scope Phase 7:**
- email, password recovery;
- user-chosen `db_path` (UI переноса БД);
- rate limiting (→ Phase 9).

**Критерий выхода:**
- Register → isolated user DB → заполнение профиля → upload + chat.
- Пользователи не видят чужие данные; logout отзывает сессию.

**Зависимости:** Phase 6.

## Phase 8 — Large KB & Document Index

**Концепция:** 
[§2.3](см. 23-chat-консультация--агентный-диалог) (абзац о росте KB и `kb.list_documents`).

**Цель:** масштабирование при десятках+ документов на пациента.

**Scope:**
- Сводка KB в system prompt (категории, диапазон дат, count) вместо полного списка.
- Инструмент `kb.list_documents` (или эквивалент) для полного каталога по запросу модели.

**Критерий выхода:**
- Старт сессии остаётся в пределах бюджета токенов при большой KB; модель может получить полный список документов через tool.

**Зависимости:** Phase 1; желательно Phase 6–7 (реальные объёмы у пользователей).

## Phase 9 — Internet access / Rollout

**Концепция:** 
[§8.3](см. 83-деплой-и-доступ-из-интернета).

**Цель:** публичный (или широкий) доступ и подключение пользователей.

**Scope:**
- TLS, публичный endpoint, rate limit, security headers.
- Onboarding через [§7](см. 7-регистрация-и-управление-учётной-записью) (username + пароль; email — опционально позже) — **обязателен** перед открытием доступа.
- Мониторинг и эксплуатация production-like.

**Критерий выхода:**
- Внешние пользователи регистрируются и работают изолированно; нет доступа к чужим данным.

**Зависимости:** Phase 7 (и Phase 6). Phase 5 — закрытый multi-user deploy, не замена Phase 9.

## Phase 10 (optional) — Image / scan ingest

**Концепция:** 
[§2.1](см. 21-ingestion-индексация-документов), 
[§6](см. 6-пользовательский-интерфейс-фронтэнд) (загрузка).

**Цель:** индексировать сканы и фото медицинских документов, не только Markdown.

**Зачем optional:** не блокирует Phase 4 (веб-MVP может стартовать с upload `.md`); для реальных данных сканы часто критичны.

**Scope:**
- Приём файлов: изображения и PDF (точный набор — при реализации).
- OCR и/или vision-LLM → текст или markdown; исходный файл в filestore.
- Интеграция в ingest saga (шаг до chunking); далее существующий pipeline.
- CLI: `ingest` принимает пути к image/PDF.
- Web: расширение формы upload (после backend), если Phase 4 уже есть.

**Критерий выхода:**
- Скан/фото → индекс в KB → находится через retrieval и [§2.3](см. 23-chat-консультация--агентный-диалог).

**Зависимости:** Phase 0; параллельно с Phase 4–5 при необходимости; не блокирует Phase 6–9.

## Вне roadmap (не в концепции)

### Dev / Smoke — `consult` (не фаза)

One-shot: жалоба → baseline retrieval → один вызов LLM → ответ.  
Назначение: быстрая проверка цепочки KB + промптов без agentic-цикла.  
Не заменяет Phase 1 и не входит в продуктовый путь по [§2.3](см. 23-chat-консультация--агентный-диалог).
